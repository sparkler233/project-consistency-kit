新对话开局,把项目当前状态完整装载进来再继续工作。按以下四层吸收上下文。

> 配套:出向命令是 `/同步`(把改动联动落盘 + 提交)。机制见 `一致性机制/机制设计说明.md`。

## 第一层:项目概览

依次完整读取:

1. `CLAUDE.md` —— 项目背景、流程、协作规则、决策记录
2. `README.md` —— 目录地图

## 第二层:项目状态(git 是事件源)

**「上次同步后发生了什么」由 git 回答**,不靠手维护的日志。

1. 看提交历史(已沉淀的事件流水):
   ```bash
   git log --oneline -20
   ```
2. 看工作区现状(**尚未提交 = 正在进行 / 上次没收尾的工作**):
   ```bash
   git status --short
   ```
3. 看未同步范围(自上次 `/同步` 以来,含已 commit 未同步的):
   ```bash
   if git rev-parse -q --verify refs/tags/synced >/dev/null; then
     d=$(git diff --stat synced); [ -n "$d" ] && printf '%s\n' "$d" || echo "(已同步:synced 已在 HEAD,无新改动)"
   else
     echo "(无 synced tag,尚未跑过 /同步)"
   fi
   ```
4. 完整读 `CLAUDE.md` / `README.md` 之外的**中枢文档**(见 `一致性机制/文件联动目录.md` 的 Part A 清单);体量大时只读最新一份的结构与状态,避免 context 浪费。

> 不是 git repo → 报「项目未纳入 git,一致性机制未生效」,退回只读现有文件来判断状态。

## 第三层:当前焦点(关键)

**「接着上次那个字继续」的关键,是第二层 `git status` 里那些未提交 / 未跟踪的文件**——它们就是上次没收尾的活。逐个**完整读取**,弄清写到哪了。

辅以提交历史里的线索定位焦点:

| 线索 | 应跟进读的文件 |
|------|--------------|
| `git status` 显示某文件被改 | 直接读那个文件 |
| 最近 commit 的 message 提到某主题 | 找该主题对应的文件 / 目录 |
| 工作区有二进制文件夹变动迹象 | 对应目录的 `_manifest.md` |

**判定规则**:

- `git status` 有未提交文件 → 直接读它们(最高优先)。
- 工作区干净但要续上一个方向 → 看 `git log` 末几条 + 对应文件。
- 完全无线索 → 跳过本层,在报告「卡点」记一笔「未识别到当前焦点文件」。

## 第四层:全景扫描(自动派生,不写死清单)

不要维护一份固定目录清单(项目目录会增减,写死的清单迟早过期)。**从仓库本身派生**:

1. 列出实际存在的顶层目录(已跟踪 + 未跟踪):
   ```bash
   # 仓库已有 commit 时列已跟踪的顶层目录(quotepath=false 让中文目录名正常显示)
   git rev-parse -q --verify HEAD >/dev/null 2>&1 \
     && git -c core.quotepath=false ls-tree -d --name-only HEAD
   ls -A                                # 含未跟踪 / 占位 / 尚无 commit 的情况
   ```
2. 对每个非隐藏顶层目录 `ls`,判断哪些**有实质内容**、哪些只是 `.gitkeep` 占位。
3. 目标是建立「项目实际形态」的认知——哪里有料、哪里是空的——而不是核对某张清单。

## 输出 catchup 报告

四层装载完后,按以下结构回复,**每节简短,3–5 行内**:

### 项目快照
一句话:项目主题 / 当前阶段(取自 CLAUDE.md)。

### 已完成
列已沉淀的关键产物(按项目自己的阶段 / 里程碑划分,取自 CLAUDE.md 的工作流)。没有就写「暂无」。

### 进行中
「上次最后在做什么」——基于 `git status` 未提交文件 + 第三层读到的焦点文件。无线索写「暂无记录」。

### 卡点 / 待决策
明确的 TODO / 开放问题(来自焦点文件里的 TODO、备注、未覆盖的联动)。没有写「无」。

### 下一步建议
2–3 个最自然的下一步,按优先级排序。

## 等指令

报告结尾问一句:

> catchup 完成。继续上次的工作,还是有新方向?

## 守则

- 不凭印象作答,所有结论来自上面读过的文件 / git 输出。
- 若 `CLAUDE.md` / `README.md` 与实际目录或 git 状态有出入(决策没记、目录变了、命令表过期),报告末尾用「⚑ 一致性提醒」标出。
- catchup 不做实际工作,只汇报状态、等用户指令。
- 偏差严重时,建议用户跑 `/同步` 把项目状态对齐。
- 仓库尚无任何 commit(刚 `git init`)时,`git log` / `git ls-tree HEAD` 会报无 HEAD——以 `ls` 结果为准即可,不影响 catchup。
