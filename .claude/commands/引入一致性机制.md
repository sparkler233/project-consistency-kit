<!-- 一致性机制 version: 2026-06-12 -->

把【项目一致性机制】装进**当前项目**。和绿地初始化(`初始化新项目.md` 的整包 rsync)不同,本命令面向**已经存在、已有自己文件 / git 历史的项目**,核心原则两句话:

> **① 增量 + 合并,绝不覆盖。** 一切碰到用户已有文件的地方都是「追加 / 合并」,先列计划、经用户确认才动手;插入前一律 grep 防重复,可安全重跑(幂等)。
> **② 适配,不假设。** 目标项目的 CLAUDE.md / README 很可能**不长我们骨架的样子**,还会有它自己的关键文档。凡是「随项目成形」的部分(纪律放哪、README 加不加表、Part A 中枢清单是什么)一律**探测 + 询问**,不套模板。拿不准就停下问,别猜。

> **套件源(只读)**:默认 `~/Developer/项目一致性机制`(本机套件仓库)。下文凡「从套件拷 / 读套件」都指它。读不到就问用户它在哪。**绝不修改套件本身。**
> 全新空项目也能用——那只是「所有目标文件都不存在」的退化情形,本命令照样跑。

---

## 步骤 0 · 前置检查 + 探测现状

1. 确认 **当前工作目录是目标项目根**,不是套件目录本身(`pwd` 若等于套件路径 → 停下提示:请在要引入机制的项目里跑)。
2. 确认套件源可读(默认 `~/Developer/项目一致性机制`,读不到就问)。
3. 探测目标项目现状,建一张「现状表」(后面计划据此分流):

   ```bash
   git rev-parse --is-inside-work-tree 2>/dev/null   # 是不是 git repo
   git rev-parse -q --verify HEAD                     # 有没有 commit
   git rev-parse -q --verify refs/tags/synced         # synced tag 是否已存在
   ls CLAUDE.md README.md .gitignore .gitattributes 2>/dev/null
   ls .claude/commands/ 一致性机制/ 2>/dev/null
   grep -hE '^(<!-- |# )一致性机制 version:' .claude/commands/*.md 一致性机制/*.md 一致性机制/hooks/*.sh 2>/dev/null | sort -u   # 已装机制文件版本(锚定行首避开正文引用;hooks/*.sh 用 # 注释格式,故并入)
   ```
   再看内容级状态(关键——决定是「合并进现有结构」还是「新建」):
   - `CLAUDE.md`:存在吗?**它的结构长什么样**(有没有「约定 / 规范 / 协作」类章节可挂?有没有「决策记录」类章节?)?是否已含同步纪律(grep `一致性机制:同步纪律` 或「同步纪律」)?
   - `AGENTS.md`:存在吗?⚠️ **若 CLAUDE.md 与 AGENTS.md 并存**——它俩常是一份拷自另一份、且其中一份已过时。**先 `diff CLAUDE.md AGENTS.md` 判断谁是当前版**(看哪份含最近 commit 的内容 / 较新的 `last-updated`),**别假设某个就是新的**(实测踩过:差点把过时副本当真身)。处理策略见步骤 3。
   - `README.md`:存在吗?是**项目地图**式,还是带 badge / 安装 / API 的**软件门面**式?是否已含机制指针 / 命令表?
   - `.claude/commands/catchup.md` / `同步.md`、`一致性机制/` 是否已存在?
   - **已装文件的版本**:上面 grep 出的 `一致性机制 version:` 行 vs 套件同名文件的——项目侧更旧、或**没有版本行**(早于版本治理的旧版)→ 该文件标 🔼 可更新(步骤 2)。

4. 判定:**全新引入** / **部分已装(更新模式)**。已装的部分后面一律标「⏭️ 已存在」,不重复装。

## 步骤 1 · 拟订安装计划 → 用户确认

把「将要做什么」汇总成一张表,每项标 **新建 / 合并追加 / 🔼可更新(版本行旧,步骤 2)/ ⏭️跳过 / ❓询问**。**先给用户看,确认后才执行**(节奏同 `/同步`)。

```
引入计划
┌────────────────────────────────┬────────────────────────────────────┬──────────┐
│ 目标                           │ 动作                                │ 状态     │
├────────────────────────────────┼────────────────────────────────────┼──────────┤
│ .claude/commands/(catchup/同步)│ 从套件新建                          │ 新建     │
│ 一致性机制/(4 文档 + hooks)  │ 从套件新建                          │ 新建     │
│ .claude/settings.json          │ 合并接线 Stop → 收尾提醒(JSON 增量)│ 合并追加 │
│ CLAUDE.md                      │ 自适应注入同步纪律 + 决策记录锚点    │ 合并追加 │
│ AGENTS.md(若有 / 用 Codex)   │ 软链 → CLAUDE.md(双文件策略,步骤3)│ ❓询问   │
│ README.md                      │ 最小侵入:加一行机制指针             │ 合并追加 │
│ 文件联动目录.md Part A         │ 发现项目真实中枢文档并登记(步骤 4) │ ❓询问   │
│ .gitignore / .gitattributes    │ 二进制策略——看项目类型            │ ❓询问   │
│ synced tag                     │ 打在当前 HEAD                       │ 新建     │
└────────────────────────────────┴────────────────────────────────────┴──────────┘
```

**必须先问用户的几件事**:
- **项目类型**:纯代码 / 纯文字,还是含图片 / 设计稿 / 字体?→ 决定步骤 5 要不要二进制规则。
- 若需**新建 CLAUDE.md**:用**极简版**(默认,软件 / 已有项目)还是**全骨架版**(创作类项目)?
- 若项目**用 Codex / 已有 AGENTS.md**:确认按双文件策略让 `AGENTS.md` 软链到 `CLAUDE.md`(步骤 3);若两者已并存,先 diff 定新旧。
- 若目标**不是 git repo**:是否允许 `git init`?(不擅自 init。)

确认方式同 `/同步`:「是 / go」全做,「跳过第 N 行」「不要 X」「取消」。

## 步骤 2 · 拷入项目无关文件(新建,不覆盖)

「直接可用、项目无关」的机器件,**不存在才拷;已存在比版本行**:项目副本的 `一致性机制 version:` 更旧、或没有版本行(早于版本治理)→ 🔼 提示「套件 <X> / 项目 <Y>,要更新吗?改了什么见套件 CHANGELOG.md」,**用户同意才**以套件版整文件覆盖;版本相同 → ⏭️ 不动。

- `.claude/commands/catchup.md`、`.claude/commands/同步.md`
- `一致性机制/机制设计说明.md`、`一致性机制/README.md`
- `一致性机制/hooks/收尾提醒.sh`(出向兜底 hook,见设计说明决策 10;拷后 `chmod +x`)
- `一致性机制/决策档案.md`(决策记录的历史档案骨架,规则 6 轮转的目的地;**已存在(有归档内容)绝不覆盖**)
- `一致性机制/文件联动目录.md`:不存在 → 拷套件骨架;**已存在**(项目可能已写自定规则)→ **绝不覆盖**,只比对通用规则 1–6 是否齐全,缺的提示补。版本行旧时也不整文件覆盖(自定区是用户的),改按套件 `CHANGELOG.md` 提示通用规则有无更新、逐条问要不要补。

> 先 `mkdir -p .claude/commands 一致性机制/hooks`,再 `cp`。

**`.claude/settings.json`(hook 接线,JSON 增量合并)**
- 不存在 → 新建,内容照套件 `.claude/settings.json`。⚠️ **用 Write/Edit 工具直接写入,别用 shell `cp`**:settings.json 属权限机制文件,`cp …/.claude/settings.json …` 这类命令会被 Claude Code 权限分类器当「自我修改权限配置」拦截(实测踩过);内容很短、对用户透明,直接写即可。
- 已存在 → **增量合并,绝不覆盖**:无 `hooks` 键 → 加键;有 `hooks` 无 `Stop` → 加 `Stop` 数组;已有 `Stop` → 把收尾提醒 entry **append** 进数组,不动用户已有 hook。
- 防重复:文件已含「收尾提醒」→ ⏭️ 跳过(JSON 无法带版本行,幂等靠 grep)。

## 步骤 3 · 合并中枢文件(自适应,绝不覆盖)

**核心:同步纪律是一段自包含文字,带 marker 注入到对方的结构里,而不是要求对方长成我们的结构。**

**先定真身:CLAUDE.md / AGENTS.md 双文件策略**(实测踩坑后补)

工具加载行为不同,直接决定纪律往哪注:**Claude Code 只读 `CLAUDE.md`**(不读 AGENTS.md);**Codex 只读 `AGENTS.md`**(AGENTS.md 是跨工具开放标准,Cursor / Jules / Amp 等也认)。规则:

1. **真身永远是 `CLAUDE.md`**:纪律注入它,catchup/同步 命令与 Part A 也引用它(套件默认,不用改)。因为本机制随 Claude Code 用,纪律必须被它**每会话读到**。
2. **项目有 / 想要 AGENTS.md**(用 Codex 等)→ 不另存一份,改 `ln -s CLAUDE.md AGENTS.md`。一份内容:Claude Code 读 CLAUDE.md、Codex 读 AGENTS.md→CLAUDE.md,两个工具都喂到。(只有 AGENTS.md、无 CLAUDE.md 时:先 `mv AGENTS.md CLAUDE.md` 再软链回去。)
3. **两者已并存且内容不同** → ⚠️ **先 `diff` 判断谁是当前版**(含最近 commit 内容 / 较新 `last-updated` 为准),当前版当真身,过时那份弃用后软链。**绝不假设某个就是新的**——实测差点把过时副本当真身、险些回退掉一次安全更新。

> Windows 不便 symlink 时:**内容仍留在 CLAUDE.md(真身不变)**,让 `AGENTS.md` 做一行指针文件(「本项目协作规则见 CLAUDE.md,请先完整读取它。」),Codex 等读到会照做。**不要**反向在 CLAUDE.md 里写 `@AGENTS.md` 导入——那会把正文逼进 AGENTS.md、CLAUDE.md 变空壳,而 `/catchup` 第一层与 Part A 中枢检查直读 CLAUDE.md 文件本身,会读到空壳。

**CLAUDE.md**
- **不存在** → 按步骤 1 选的建法新建:
  - **极简版(默认)**:项目名 + 一句话说明 + `## 工作约定`(内含同步纪律)+ `## 关键决策记录`(一行占位)+ 指向 `一致性机制/`。**不**套用创作骨架的「相关方 / 体量 / 创作模式」表。
  - **全骨架版**:拷套件 `CLAUDE.md`(适合创作类项目)。
- **已存在** → 把套件 `CLAUDE.md` 里 marker 包裹的「`### 同步纪律(git 驱动)`」整块原样注入(套件里它已带 marker,begin 行含套件版本):
  ```
  <!-- 一致性机制:同步纪律 begin (version: <套件版本日期>) -->
  ### 同步纪律(git 驱动)
  …(套件该节正文)…
  <!-- 一致性机制:同步纪律 end -->
  ```
  - **放哪**(按语义就近,不强求章节名):① 有「约定 / 规范 / 协作 / 规则」类章节 → 挂其下;② 没有 → **末尾新建**一节 `## 一致性机制 · 同步纪律` 再包进去。
  - 已含 marker → 看 begin 行的 version:与套件相同 → ⏭️ 跳过;更旧或无 version(旧版注入)→ 🔼 询问是否**整块替换** begin/end 之间内容为套件新版(只动块内,块外一字不碰)。
  - 无 marker 但 grep 到「同步纪律」(手写或上古版本)→ ⏭️ 跳过,提示人工核对。
  - **决策记录锚点**(用**标题 grep** 定位,**有意不像纪律块那样用 marker 包裹**——决策记录是 append-only 的用户内容、**永不整块替换**,故不带 version marker;模板决策节同样无 marker,两者一致):grep「关键决策记录」/「决策记录」;没有 → 末尾追加 `## 关键决策记录` + 轮转说明行(只留最近 ~10 条,超限由 /同步 轮转到 `一致性机制/决策档案.md`——规则 6)+ 一行 `- YYYY-MM-DD:启用一致性机制`。

**README.md(最小侵入)**
- **不存在** → 拷套件最小地图骨架。
- **已存在** → **默认只追加一行 marker 指针**,不重排对方 README:
  ```
  <!-- 一致性机制:指针 begin -->
  > 🔄 本项目启用了「一致性机制」(catchup/同步 自我维护):新会话跑 `/catchup` 装载状态,收尾跑 `/同步` 落盘提交。机制全貌见 `一致性机制/`。
  <!-- 一致性机制:指针 end -->
  ```
  放在标题下第一段之后(或文件末尾)。已含指针(grep marker)→ ⏭️ 跳过。
  - 完整命令表说明本就在 `一致性机制/README.md` 里,不往项目 README 塞表。
  - **例外**:用户明确要在项目 README 里放完整 `/catchup`、`/同步` 命令表(opt-in)→ 才加。

> 所有插入前先 grep marker / 关键词,命中即跳过——保证重跑不产生重复段落。

## 步骤 4 · 发现并登记中枢文档(Part A 的灵魂)

> 机制的联动价值**全靠 Part A 反映项目真实的关键文档**。只装通用 4 条(CLAUDE/README/机制文档),联动对该项目自己的 ADR / spec / 契约 / 大纲就是**半瞎**的。这步把「假设 Part A」变成「发现 Part A」。

1. **扫描候选**(只在已跟踪 + 现存文件里找,别臆造):
   ```bash
   git -c core.quotepath=false ls-files 2>/dev/null   # 已跟踪文件
   ls -A; ls docs doc 文档 设计 2>/dev/null            # 顶层 + 常见文档目录
   ```
   挑出**像关键文档**的:根目录除 README/CLAUDE 外的 `*.md`;`docs/` 下的设计 / 规范;命名含 `ADR / decision / spec / schema / contract / openapi / .proto / design / RFC / 设计 / 规范 / 契约 / 大纲 / 风格` 的文件。
2. **列候选问用户**,每条附一句「为什么像中枢 + 它该随什么改动而更新」的猜测:
   ```
   发现这些疑似关键文档,纳入 Part A 中枢清单吗?(中枢清单宜保持 4–10 个,变化慢、好枚举)
   - docs/api-contract.md   → 像 API 契约;接口增改时下游要对齐
   - docs/adr/             → 像架构决策记录;重大技术决策落点
   - 风格定义.md            → 像视觉规范;锁定后回扫配图
   选哪些?也可补我没扫到的。
   ```
3. 用户确认 / 增删后,把选中的写进**该项目的** `一致性机制/文件联动目录.md` 的 **Part A 表**(每条:文档路径 + 「它应随什么改动而更新」)。grep 防重复。
4. 顺带问一句:**有没有稳定的领域联动规则**(如「改 A 必回扫 B」)要写进 Part B 的「项目自定规则区」?有就按其格式登记,没有就跳过。

> 拿不准某文档算不算中枢 → 列出来让用户定,别替他决定。

## 步骤 5 · 二进制策略(按项目类型,追加合并)

- **含图片 / 设计稿 / 字体**:把套件 `.gitignore` / `.gitattributes` 相应规则**追加**进目标(已有同名文件**只补缺的行,不删用户已有行**;没有就整份拷)。用栅格图 LFS 的,提示首次 `git add` 前跑 `git lfs install --local`。
- **纯代码 / 纯文字**:跳过二进制规则,或只补 `.gitignore` 的「系统 / 工具文件」区(`.DS_Store` 等)。问一句即可。

> `.gitignore` / `.gitattributes` 一律**追加合并**,绝不覆盖用户已有内容。

## 步骤 6 · 立 synced horizon

- **不是 git repo**:经步骤 1 同意后 `git init && git branch -M main`;否则停下提示「未纳入 git,机制无法运行」并跳过本步。
- **有 commit** → `git tag -f synced`(打在当前 HEAD——机制从现在起算,既往历史不追溯联动)。
- **无 commit**(空仓):提示「需先有一次 commit 才能立 horizon」,**询问**是否帮忙 `git add -A && git commit`(message 由用户确认),commit 后再 `git tag synced`。不擅自 commit。
- `synced` **已存在** → 不动(机制此前已装过),⏭️。

## 步骤 7 · 报告 + 下一步

```
引入完成。
- ✅ 新建:N 项(列出)
- 🔀 合并追加:M 项(CLAUDE.md +同步纪律 / README +指针 / Part A +K 条中枢…)
- ⏭️ 跳过(已存在):K 项
- ❓ 待你处理:Q 项(填 CLAUDE.md 占位 / 补 Part B 自定规则…)
- 📌 synced horizon 已立在 <短 hash>
```

收尾提示:
- 若新建了 CLAUDE.md:去填项目背景 / 约定的占位。
- 复核 Part A 中枢清单是否齐(漏了的关键文档随时补)。
- **单机约定**:`/同步` 只到本地 commit,**push 由你手动**(`git push`)。
- 首次会话 Claude Code 会请求确认项目 hooks(收尾提醒),确认一次即可。
- 跑一次 **`/catchup`** 验证装载正常。

## 守则

- **绝不覆盖**用户已有内容;凡碰现有文件都是合并 / 追加,且必须在步骤 1 计划里展示、经用户确认。
- **适配不假设**:目标 CLAUDE.md / README 结构与我们骨架不同是常态——用 marker 注入自包含块、按语义就近放置,**不重排对方文件、不强加我们的章节结构**。
- **中枢文档靠发现不靠默认**:Part A 必须含项目自己的关键文档,这些只能扫描 + 问用户,不能套模板。
- **CLAUDE.md 是唯一真身**:纪律注入 CLAUDE.md(Claude Code 只读它),要喂 Codex 等就让 `AGENTS.md` 软链过去、不另存第二份;两者并存且不同 → 先 `diff` 定新旧,**别假设**。
- 一切插入**先 grep marker / 关键词防重复**——marker 同版命中即跳过(⏭️),版本旧按步骤 2/3 询问更新(🔼);本命令可安全重跑(幂等)。
- **不擅自** `git init` / `commit` / `push`——涉及这些都先问。
- `.gitignore` / `.gitattributes` 只追加缺的行,不删用户已有行。
- **套件源只读**,绝不修改套件本身。
- 这是兜底安装器:任何分歧点(结构怪、文档树不认识、算不算中枢)**停下来问,别猜**。
