# 一致性机制 —— catchup / 同步 自我维护系统

> 本目录把「项目的自我维护机制」(会话切换、上下文加载、改动落盘)和「项目本身的内容」分开收纳。
> **想了解或修改这套机制,从这里看起。**

## 这是什么

一套让 AI 协作能跨会话连续工作、并自动维护文件一致性的系统:

- **入向**:新会话用 `/catchup` 把项目状态加载回来;
- **出向**:收尾用 `/同步` 按联动规则把改动落盘、提交、推进 `synced` 标记;
- **底座**:git 是唯一事件源,二进制走选择性处理(可选 LFS)。

完整设计动机见 [`机制设计说明.md`](机制设计说明.md)。

## 本目录里有什么

| 文件 | 作用 |
|------|------|
| `README.md`(本文件) | 机制索引 + 登记钉在别处的文件 |
| [`机制设计说明.md`](机制设计说明.md) | 整套设计的「为什么」:每个决策的动机与取舍。给人 / 接手 AI 看 |
| [`文件联动目录.md`](文件联动目录.md) | 联动规则(中枢文档 + 例外)。`/同步` 运行时读取它 |

## ⚠️ 这些文件也属于本机制,但被工具钉死在别处(不能移进来)

| 文件 | 实际位置 | 为什么不能进 `一致性机制/` |
|------|----------|----------------------|
| `/catchup` 命令 | `.claude/commands/catchup.md` | Claude Code 只在 `.claude/commands/` 找斜杠命令,移走就失效 |
| `/同步` 命令 | `.claude/commands/同步.md` | 同上 |
| 二进制排除规则 | `.gitignore`(仓库根) | git 要它在根才全局生效 |
| 选择性 LFS 规则 | `.gitattributes`(仓库根) | git/LFS 要它在根才全局生效 |
| 同步纪律 | `CLAUDE.md` 的「同步纪律」节 | CLAUDE.md 必须在根、每次会话自动加载;纪律要被每轮读到,埋进 `一致性机制/` 就没人读 |
| sync horizon | git tag `synced` | git 对象,不是文件 |

> 换句话说:**本机制在文件层面天生和项目缠在一起,是工具决定的,不是没设计好。** 本目录已尽可能把「能搬的文档」聚拢;上表是「搬不动的那几个」的登记册。改机制行为时,记得这几处也要一起看。

## 在新项目里启用本机制

- **绿地(全新项目)**:整包套用,完整步骤见模板仓库根的 `初始化新项目.md`(随模板分发,开张后通常已删)。
- **已有项目(已有文件 / git 历史)**:别整包拷,用全局命令 **`/引入一致性机制`**——增量 + 合并、绝不覆盖,并会扫描登记该项目自己的中枢文档。

## 改了机制怎么办

机制自己的文件变动也走同一套流程:改完跑 `/同步`,它会按 [`文件联动目录.md`](文件联动目录.md) 检查要不要联动更新 `CLAUDE.md`、`README.md` 等,再 commit、推进 `synced`。
