# 文件联动目录

<!-- 一致性机制 version: 2026-06-18 -->

> 本文件是「联动规则层」。定义:**某个文件变了,还要跟着检查 / 改哪些其他文件。**
> `/同步` 命令读本文件来推断联动动作。
>
> 设计动机见 [`机制设计说明.md`](机制设计说明.md),命令逻辑见 [`.claude/commands/同步.md`](../.claude/commands/同步.md)。

## 怎么用这份目录

`/同步` 拿到「上次同步以来的所有改动」后,对每个变化的文件:

1. 先比对 **Part A · 中枢文档清单** —— 这个改动是否影响了某份中枢文档的内容?
2. 再比对 **Part B · 例外规则** —— 改动路径是否命中某条特定规则?
3. 两者都没命中的,靠 AI 临场判断是否还有隐性联动(这是兜底,不是保证——见设计说明决策 2 的「代价」)。

> **双层模型的边界**:Part A 保证「最重要的几个文档总会被检查」;Part B 显式化「AI 反复漏判的机械规则」。其余靠判断。新发现的稳定联动 → 补进 Part B。

---

## Part A · 中枢文档清单

这些是项目的「关键文档」。每次 `/同步`,AI 都要主动检查本轮改动**是否需要回头更新它们的内容**。

| 中枢文档 | 它应随什么改动而更新 |
|----------|--------------------|
| `CLAUDE.md` | 拍板了决策、改了流程 / 工作约定 / 协作分工、项目背景变化(范围 / 目标 / 规格) |
| `README.md` | 目录结构变了、斜杠命令增删或状态变化 |
| `一致性机制/文件联动目录.md`(本文件) | 出现了新的稳定联动关系 |
| `一致性机制/机制设计说明.md` | 一致性机制本身发生真实变化 |
| 〔项目自定中枢文档〕 | 〔在此追加你项目的关键文档——见下方说明〕 |

> 中枢清单应保持在 4–10 个。变化慢、好枚举。加新中枢文档时在此追加一行。
>
> **项目自定示例**(供参考,按需替换):一本书项目会加 `docx/策划/ 下的大纲`(随章节增删更新)、`风格定义.md`(随视觉风格锁定更新);一个软件项目可能加 `API 契约文档`、`架构决策记录(ADR)`。

## Part B · 例外规则

只列「AI 容易漏判」或「机械化、应固定执行」的特定联动。下面 1–6 条是任何项目都适用的通用规则;项目专属规则写在末尾的「项目自定规则」区。

### 规则 1 · 机制件变化(命令 / hook / 接线)→ README + CLAUDE + 机制文档

**触发**:`.claude/`(commands/ 或 settings.json)或 `一致性机制/hooks/` 下有文件新建 / 删除 / 改动
**动作**:
- `README.md` 的「斜杠命令」表同步更新(新增 / 状态 / 描述);
- 若是**新命令首次出现**且 `CLAUDE.md` 的工作约定 / 协作分工有引用关系,一并对齐;
- 若是 **hook 脚本 / 接线**变化:对齐 `一致性机制/README.md`(目录表 + 钉死登记册)与 `机制设计说明.md`(对应决策节)。

### 规则 2 · B 类决策落地的 canonical 落点

**触发**:safety net 捕获到一条**决策**(B 类事件)
**动作**:决策的**规范落点是 `CLAUDE.md` 的「关键决策记录」**(一行:`- YYYY-MM-DD:<摘要>`)。
- 若项目另有进度纪要 / 决策日志,可同时补一条;无则跳过。
- **防重复**:写入前 grep 日期 + 摘要前缀,已存在则跳过。
**为什么钉死落点**:避免「决策有时落 CLAUDE.md、有时落别处、有时两边都没有」的不一致(设计说明决策 3)。

### 规则 3 · 产物目录内容增删 → 更新该目录的状态描述

**触发**:某个「产物目录」里新增 / 删除了文件
**动作**:更新该目录 README / 状态清单里的**文字描述**(如某目录 README 的「当前已有」表、状态总览)。
**为什么需要**:git 看得见文件增删,但**不会自动改 README 里那句文字状态**——状态描述会单方面过期(「空,待产出」明明已经有产物了)。

### 规则 4 · 收到外来二进制 → 写 .md 摘要

**触发**:项目里出现**外来的** `.docx` / `.pdf` / 其他二进制(对方发来的、git 看不见内容的)
**动作**:AI 写一份 `.md` 摘要放在同目录,git 跟踪摘要而非原件。原件作为附件存在,不期望 diff。

### 规则 5 · 素材 / 大文件目录内容变 → 更新 manifest(条件触发对账)

**触发(条件)**:**仅当本轮 `git diff synced` 显示改动落在素材 / 大文件目录**(任何含二进制的目录,或 LFS 指针变了)。纯文字同步不触发本规则。
**动作**:`ls` 相关文件夹、与其 `_manifest.md` 对比,把差异计入待办;更新 manifest 的 **用途 / 来源 / 授权或版权**(有美术 / 设计方向的项目可再加一项:风格基准引用)。
**manifest 只记什么**:用途、来源、授权 / 版权——这些是 git/LFS 不管的(有视觉 / 设计方向的项目可再记一项「风格基准引用」)。**不记**文件清单 / 增删 / 版本(栅格图归 LFS,gitignore 的归 `ls`)。
**为什么仍需要**:① 被 `.gitignore` 排除的二进制(设计源 / 字体)git 看不见,靠 `ls` 对账兜底;② 授权 / 版权是 LFS 永不记录、却往往必需的信息。
**为什么条件触发**:manifest 对账若每次同步都跑,绝大多数(纯文字活)是空转,只会给同步流程加「可靠性税」。只在真动了素材时跑,既保安全网又零额外负担。
**范围**:纯文档 / 文稿目录不要 manifest(git 完美跟踪 .md)。

### 规则 6 · 决策记录超限 → 轮转归档(条件触发)

**触发(条件)**:`/同步` 时 `CLAUDE.md`「关键决策记录」条目**超过 10 条**。未超不触发,零成本。
**动作**:把轮转列入待执行计划(经用户确认,同其他联动):
- 把**最老的**条目**剪切**到 `一致性机制/决策档案.md` 末尾(保持时间升序、原行原样;文件不存在则按套件骨架新建),CLAUDE.md 只留**最近 10 条**;
- 确保 CLAUDE.md 该节有指针(「更早的见 `一致性机制/决策档案.md`」),没有则补一行。
**为什么**:CLAUDE.md 每轮对话自动进上下文,每行都在交「token 租金」;决策记录是唯一 append-only 的节,不轮转迟早变档案馆(设计说明决策 11)。档案**不进 Part A**、`/catchup` 不默认读——历史按需查,不上每轮加载的车。
**防重复**:轮转 = 剪切而非复制;档案里已有同日期同摘要的行 → 跳过该行并提示人工核对。

## 添加新规则的格式

```
### 规则 N · <一句话规则名>

**触发**:<什么文件 / 什么情况>
**动作**:<联动到哪、做什么、是否自动改、防重复策略>
```
