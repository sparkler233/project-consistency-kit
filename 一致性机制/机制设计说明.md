# 项目一致性机制 · 设计说明

<!-- 一致性机制 version: 2026-06-11 -->

> **本文档受众**:接手某个用了本机制的项目的 AI 协作者,或想理解这套自我维护机制的人。
>
> **本文档目的**:说明机制的设计、每个设计选择背后的"为什么",以及取舍时考虑过的边角问题。
>
> **一句话**:本机制让 AI 协作的项目能**跨会话不走样**——新会话能完整接上上次,改动能系统落盘,文件之间的隐性依赖不会慢慢腐烂。

---

## 一、它解决什么问题

任何持续演进的项目,文件之间都会长出大量**隐性联动**:改了 A,B/C/D 理应跟着改,但没人记得。

| 一个改动 | 其他理应跟着改的地方 |
|----------|--------------------|
| 拍板一条重大决策 | 决策记录、相关方案文档、对外口径 |
| 改了某个中枢文档 | 引用它的下游文件、状态清单 |
| 新增/删除一类产物 | 目录的状态描述、地图、索引 |
| 与外部对接 | 沟通记录、可能受影响的计划 |

靠人或 AI 的注意力维护这些联动,**漏改是迟早的事**。漏改的后果不是立刻爆炸,而是慢慢腐烂——某天你以为某个口径定在 A 文件,结果 B、C 还停在两版前。本机制就是专门兜底这件事的。

它还顺带解决另一个问题:**会话不连续**。AI 协作常常一个会话做一半,新会话开局对"上次做到哪了"一无所知。本机制用 git 当"项目记忆",让新会话一条命令就装载回完整状态。

## 二、机制总览(三层 + 一条纪律)

```
┌─────────────────────────────────────────────────────────┐
│ git 层 —— 事件源                                          │
│  项目本身是 git repo。"发生了什么"由 git diff/log 回答,    │
│  零纪律、自动捕捉、附带完整版本史。                          │
└──────────────────┬──────────────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────────────┐
│ 联动规则层 —— 一致性的"图纸"                               │
│  一致性机制/文件联动目录.md                                │
│  "中枢文档 + 例外规则" 双层模型:                           │
│  规定"某文件变了,还要回头检查/改哪些其他文件"。              │
└──────────────────┬──────────────────────────────────────┘
                   ▼
┌─────────────────────────────────────────────────────────┐
│ 命令层 —— 入向 / 出向两个动作                              │
│  /catchup(入向):新会话把项目状态完整装载回来              │
│  /同步  (出向):按联动规则把改动落盘、提交、推进 synced     │
└─────────────────────────────────────────────────────────┘

  + 一条纪律(写在 CLAUDE.md):B 类事件落地
    决策、对外对接、口头约定这些"发生在对话里、git 看不见"的事,
    AI 必须立刻写进目标文件 —— 把"对话里的事"变成"git 能看见的改动"。
```

### `/同步` 的步骤(0 前置检查 + 5 步主流程)

```
0. 前置检查:不是 git repo / 联动目录不存在 → 退出
1. 自查 B 类事件(safety net):回看对话,找没落到文件的决策/对接 → 提示补写
2. 确定范围:git diff synced(自上次已同步的提交以来的全部改动)
3. 查联动目录:对每个改动,比对 Part A 中枢清单 + Part B 例外规则,推断联动
4. 拟订计划 → 用户确认 → 逐项执行
5. git add → 用户确认 commit message → commit → git tag -f synced(推进 horizon)
```

### `/catchup` 的四层

```
第一层 项目概览:读 CLAUDE.md(大脑) + README.md(地图)
第二层 项目状态:git log / git status / git diff synced —— 上次做到哪了
第三层 当前焦点:读 git status 里未提交/未跟踪的文件 —— 上次没收尾的活就在那里
第四层 全景扫描:自动派生顶层目录清单并 ls —— 建立对项目实际形态的认知
```

## 三、关键设计决策与"为什么"

下面逐条解释每个选择——做了什么、为什么这样选、代价是什么。这些理由是机制的精华,换任何项目都成立。

### 决策 1 · 用 git 当事件源

**做了什么**:项目纳入 git;`/同步` 用 `git diff` 知道本轮发生了什么。

**为什么**:
- 让 AI 边干边手维护一份"事件日志",依赖纪律,**会忘**(这是实践中反复观察到的弱点)。
- git 自动捕捉所有文件变化,**零纪律**。
- 顺便拿到完整版本控制(diff、回滚、历史追溯)。
- 概念干净:**"事件 = git 看见的文件变化"**。

**代价**:B 类事件(决策、对接——发生在对话里,不直接产生文件改动)git 看不见 → 需要"立刻写进文件"的纪律配合(决策 3)。

### 决策 2 · 联动规则用"中枢文档 + 例外"模型

**做了什么**:联动目录分两部分——
- **Part A 中枢清单**:列 4–10 个"项目关键文档",`/同步` 时主动检查每个改动是否影响它们。
- **Part B 例外规则**:只列"AI 容易漏判"或"机械化应固定执行"的少数特殊联动。

**为什么不穷举每种联动**:
- 项目一大,穷举式规则会膨胀失控。
- 加新文件类型时,穷举式必须同步加规则,**忘加 = 漏覆盖**(消极错误,难发现)。
- 中枢文档变化慢、好枚举;例外规则按需补,自然收敛到"刚好够用"。
- 其余靠 AI 临场判断填补。

**代价**:AI 判断可能漏。靠两层兜底——中枢清单保证"最重要的几个文档总会被检查",例外清单显式化"AI 反复漏判的具体规则"。新发现的稳定联动 → 补进 Part B。

### 决策 3 · B 类事件 safety net

**问题**:决策、对外对接、口头约定这些事发生在对话里,本身不产生文件改动,git 看不见 → `/同步` 会漏。

**解决**:`/同步` 第一步专门干这件事——回看本次对话,找"已发生但还没写进文件的 B 类事件",列清单问用户"要补写吗?",确认后立刻写入目标文件,于是 git 就能看见了。

**canonical 落点钉死**:决策的规范落点 = `CLAUDE.md` 的「关键决策记录」(联动目录有对应规则)。**钉死单一落点**,避免"一条决策有时落这、有时落那、有时两边都没有"的不一致。

**局限**:safety net 只够得到**当前会话**的对话。跨会话未落盘的 B 类事件捞不回——所以**结束会话前建议跑一次 `/同步`**。若实践中确实丢过(忘跑 `/同步` 就结束了会话),可考虑给项目配 Claude Code 的 **Stop hook** 检测未同步改动并提醒——经权衡暂不实装,保持「兜底不门禁」(决策 6)的气质。

### 决策 4 · 二进制按"值不值得留版本史 × 体积"分类处理

git 对二进制做不到 diff,且不做 delta 压缩(每版完整存储,一张 5MB 图改 10 版 = 仓库里 50MB)。所以不能无脑全进 git。策略按类:

| 类别 | 例子 | 处理 | 理由 |
|------|------|------|------|
| **文本/矢量** | `.md .txt .json .csv .svg` | 普通 git | 能逐行 diff,天生适合 git |
| **栅格图** | `.png .jpg .jpeg .webp .gif .tif .tiff .bmp .heic` | **Git LFS**(进 git,有版本史) | 值得版本化、数量多、体积中等;LFS = 指针进 git、字节进单独存储,既留版本史又不撑爆仓库 |
| **设计源/PDF/Office/字体/媒体** | `.psd .ai .indd .pdf .docx .ttf .mp4` | `.gitignore` + `_manifest.md` | 单文件可达数百 MB(吃 LFS 额度),或可替换/外来/终端产物 |
| **试稿(任意二进制)** | `_drafts/` 内 | `.gitignore` 整体忽略 | 迭代废版不进 git、不占 LFS 额度——成本控制的主开关 |

> **这是默认值,按项目可改**:纯代码/纯文字项目可能根本没有要版本管理的图片——那就清空或删掉 `.gitattributes` 的 LFS 行,本机制其余部分照常工作。规则见各项目根的 `.gitignore` / `.gitattributes`。
>
> **破例纳入某个被忽略的二进制,必须两步**:先 `git lfs track "<该文件路径>"`(把它纳入 LFS),再 `git add -f`。`git add -f` 只绕过 gitignore,**不会**自动走 LFS——少了前一步,几百 MB 的文件会按完整 blob 进普通 git、每改一版全量再存,正是本决策要防的事故。项目若未启 LFS(如删了 `.gitattributes` 的纯文字项目),先 `git lfs install --local`;只是"想留着"、不需要版本史的二进制,走 manifest 登记即可,不必破例进 git。

**被 gitignore 的二进制目录靠 `_manifest.md` 翻译**:git 看不见这些目录的内容,放一份 `_manifest.md`(纯文本)记录 git/LFS 不管、但又重要的信息——**用途 / 来源 / 授权或版权**(有视觉 / 设计方向的项目可再加「风格基准引用」)。manifest **不记**文件清单/增删/版本(那些归 git/LFS 或 `ls`),只记这几样。**只在含二进制的素材/大文件目录铺**,纯文档目录不要。

### 决策 5 · 草稿双轨(仅二进制适用)

二进制反复迭代会堆积"v1/v2/v3",若都进版本管理会污染、且吃 LFS 额度。约定:试稿放 `_drafts/`(整体 gitignore),定稿后挪出 + 让 git/LFS 接管。

**为什么只对二进制**:Markdown 每次改动本身就是一次清晰的 git 版本,不需要"草稿/定稿"区分;二进制才有"看不见的迭代"问题,需要隔出一个非正式工作区。

### 决策 6 · `/同步` 是兜底,不是门禁

用户可以随时手工改文件、手工 `git add` + `commit`。`/同步` **不是**"必须跑才能保存",它只是把"漏的联动"系统性补上、把改动收尾。重要节点(多条决策、口径更新)建议跑一下,平时想跑就跑。

### 决策 7 · commit 由用户确认,不自动提交

commit 会推进 sync horizon(下次 `/同步` 视之前为"已处理"),是个单向动作。所以 `/同步` 把联动改动写完后,AI 起草一句话 message,**问过用户**再 `commit`。

**push 同理由用户负责**:`/同步` 只到本地 commit + 推进本地 `synced` 为止,**不自动 push**。本机制按**单机**设计——云端 remote 只作备份,何时 push 由用户决定。多机协作要留意一个坑:`synced` 是本地 tag,`/同步` 用 `git tag -f` 移动它后远端不会自动跟随,需 `git push --force origin synced`,否则另一台机器的 `git diff synced` 基准会停在旧位置、漏掉联动检查。

### 决策 8 · sync horizon 用 `synced` tag,不用裸 HEAD

"上次同步到哪了"这个边界,**不能**用"最近一次 commit / HEAD"。因为决策 6 允许用户随时手工 commit:一旦用户没跑 `/同步` 就自己 commit 了,那笔改动进了 HEAD,下次 `git diff HEAD` 看不到它 → **它的联动被永久跳过,且无任何信号**。

所以 horizon = 独立的 git tag `synced`(上次**已同步**的提交)。`/同步` 第 2 步用 `git diff synced`(覆盖"已 commit 未同步"+"未 commit"两类),第 5 步末尾 `git tag -f synced` 推进。手工 commit 不再隐形。

### 决策 9 · 套件版本治理:统一版本行 + CHANGELOG

**做了什么**:每个机制文件(两条命令、安装器、本目录三份文档)头部带一行 `<!-- 一致性机制 version: YYYY-MM-DD -->`。版本是**套件级**的:任何机制文件发生真实变化,**全部版本行一起 bump 到当天**,并在套件根 `CHANGELOG.md` 记一行。`/引入一致性机制` 以版本行机械判定项目副本新旧;注入各项目 CLAUDE.md 的同步纪律块,其 begin marker 也带 version,同理可判。

**为什么**:本机制以"复制"方式分发进 N 个项目。没有版本身份,各项目里的副本会静默漂移、且无从判定谁旧——"慢慢腐烂"恰好发生在机制自己身上。"疑似旧版"必须可机械判定(一条 grep),更新提示才不是空话。版本用日期不用 semver:文档机制没有兼容性矩阵,"哪天的版"就够了。

**代价**:改机制文件多一道手续(bump 全部版本行 + 记 CHANGELOG),靠第七节的维护清单兜住。版本是套件级、非单文件级——某文件"没改也被 bump"是预期行为,换来"这个项目装的是哪版"只有一个答案。

## 四、文件类型与 git 兼容度速查

| 类型 | 例子 | git diff | 处理 |
|------|------|---------|------|
| 文本 | .md .txt .json .yaml .csv .html .css .js | 🟢 完美 | 普通 git |
| 矢量 | .svg | 🟢 是 XML | 普通 git |
| 栅格图 | .png .jpg .webp .heic | 🔴 二进制 | Git LFS(默认) |
| 设计源 | .psd .ai .indd | 🔴 + 通常很大 | gitignore + manifest |
| Office/PDF | .docx .xlsx .pdf | 🔴 二进制 | gitignore + .md 摘要 |
| 字体/媒体 | .ttf .mp4 .mp3 | 🔴 二进制 | gitignore + manifest |

**核心原则**:所有"决策、文字、规范、记录"都用 .md;需要版本史的栅格图走 LFS;其余二进制当"附件",由 manifest 翻译。

## 五、项目套用本机制后的结构

一个采用了本机制的项目,大致长这样(`<...>` 是项目自己的内容):

```
<某项目>/
├── .git/                     版本控制本体(事件源)
├── .gitignore                二进制排除规则(可按项目改)
├── .gitattributes            选择性 LFS 规则(可按项目改/删)
├── CLAUDE.md                 项目大脑 ⭐ 中枢(含同步纪律)
├── README.md                 项目地图 ⭐ 中枢
├── 一致性机制/                ★ 本机制:文档 + 规则
│   ├── README.md             机制索引 + 钉在别处的文件登记册
│   ├── 机制设计说明.md         ← 本文件
│   └── 文件联动目录.md         ⭐ 联动规则(双层模型,/同步 读取)
├── .claude/commands/
│   ├── catchup.md            入向:项目 → AI
│   └── 同步.md                出向:AI → 项目 ⭐
└── <项目自己的目录>            正文/代码/素材/产物……
```

> 注意:`.claude/commands/`、`.gitignore`、`.gitattributes`、CLAUDE.md 的同步纪律节,因工具/git 的要求**必须钉在固定位置**,搬不进 `一致性机制/`。哪些钉在哪、为什么,见 `一致性机制/README.md` 的登记册。

## 六、给接手 AI 的建议

接手一个用了本机制的项目,按此顺序读:

1. **`CLAUDE.md`** —— 项目背景、流程、协作规则、关键决策
2. **`README.md`** —— 目录地图
3. **本文件** —— 理解机制的设计动机
4. **`一致性机制/文件联动目录.md`** —— 具体联动规则
5. 跑 **`/catchup`** —— 拿当前项目状态

接手第一件该做的事:跑 `/catchup` 确认理解准确。

若对机制设计有疑问或觉得有更好方案,**先和用户讨论,别直接动架构**——每个看似奇怪的选择背后都有具体权衡,本文第三节就是解释这些"为什么"的。

## 七、本机制自身的维护

- **改任何机制文件(两条命令 / 安装器 / 本目录三份文档)= 一次套件发版**:全部 `一致性机制 version:` 行一起 bump 到当天 + 套件根 `CHANGELOG.md` 记一行(决策 9)。版本行是 `/引入一致性机制` 更新检测的依据——忘 bump,存量项目就永远收不到更新提示。
- 机制发生**真实变化**时(不只规则微调),更新本文件。
- 某个决策被推翻时,**保留原描述 + 标注"已废弃"**,不要直接删——留演进路径给接手者看。
- 改本机制的文件也走 `/同步`:改完跑一次,它会按 `文件联动目录.md` 检查要不要联动更新 `CLAUDE.md` / `README.md` 等,再 commit、推进 `synced`。
